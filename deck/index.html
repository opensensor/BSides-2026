What is firmware? What is a SoC?

// the terrain we're about to map
System on Chip

CPU + RAM + peripherals on one die

⤢ click to expand
  • ISP, NPU, crypto engines, GPIO, video encoders
  • Contains a boot ROM — immutable mask ROM
  • OTP fuses carry per-chip secrets & config
  • What runs first is fixed in silicon at fab time — not by you.
Firmware

Persistent software that runs the device itself

  • Lives on flash chips — SPI NOR, SPI NAND, eMMC
  • Layered System: bootloader → kernel → rootfs
  • Readable, modifiable, reflashable — usually
  • Its what actually runs when you plug the thing in

$ man flash --types § 02 · flash storage
Know your flash

NOR vs NAND — not the same chip

// the chip you clip onto changes everything downstream
SPI NOR
byte-addressable
⤢ click to expand
  • Small — 8, 16, or 32 MB typical
  • Execute-in-place (XIP) capable
  • No bad blocks — what you read is what's there
  • Filesystem: SquashFS or raw blobs
SPI NAND
page / block-addressable
⤢ click to expand
  • Big — 128 MB to multi-GB
  • Read in pages, erase in blocks
  • Born with bad blocks — and grows more over time
  • Filesystem: UBI + UBIFS over MTD
More space — more abstraction
NAND density is paid for in software: the kernel runs UBI — wear leveling, bad-block tracking, ECC — underneath UBIFS. Your filesystem is layers above raw silicon.
Why it matters to us
Dumping NAND is not dumping a filesystem — you get raw pages with OOB ECC bytes. Reconstructing the rootfs means decoding UBI volumes before you ever see /etc.

Everything is a file

Linux exposes hardware as nodes under /dev. Userspace never touches a register — it just opens a file and makes syscalls.

The driver's job

Translate those syscalls into register pokes, service interrupts, move buffers via DMA, and manage clocks & power — all in kernel space.

Full circle

This is the Linux Device Drivers book from slide 2, twenty years later — except now the driver we need ships as a closed blob.

→ For an IP camera, the driver that matters most is the ISP. Get it right and you have video; without it the SoC is a brick with a lens. So why can't we just write one?

How we actually reverse it

Methodology — smartdiff, LLMs, and a tracer

// the OEM blob is the spec · smartdiff narrows the search · the silicon is the judge
01 · Decompose

smartdiff MCP over HLIL

Point smartdiff at OEM tx-isp-t31.ko vs our build via Binary Ninja MCP. Get a ranked list of changed functions and function-by-function HLIL — the search is bounded before a human reads anything.

02 · Implement

LLMs write the equivalent

Feed the HLIL + behavioral delta to the model; it drafts the clean-room C against the vendor 3.10.14 tree. libimp.so stays untouched — ABI parity is the only acceptance test.

03 · Verify

Tracer driver vs OEM hardware state

In-tree driver/tx-isp-trace.c + ftrace/kprobes captures every MMIO write, ioctl arg, and IRQ. Diff our register state against the OEM trace — the silicon votes on whether we got it right.

04 · Reality

A tall headwall before first frame

Time-intensive. Months of grinding before activation and a streaming image at all — and image quality is still being chased (anchor: last "crisp image" commit, block mask 0xDD04).

→ smartdiff narrows the search · the LLM does the typing · the tracer keeps us honest. The OEM blob's runtime behavior is its de facto spec.

Tools of the trade

Reverse engineering kernel modules

// MIPS · stripped symbols · undocumented register banks
Binary Ninja

Strong MIPS HLIL, scriptable, fast

  • HLIL (High-Level Intermediate Language) reads close to C — great for ioctl dispatch tables
  • Python plugin API for batch annotation
  • Type system that survives across binaries
  • Commercial — license cost is the friction
Ghidra

Free, NSA-blessed, decompiles MIPS

  • P-code IR — cross-arch queries
  • Strong batch / headless mode
  • Decent kernel-module loader after some setup
  • Slower UX, but unbeatable price
What helps most
Loading the matching vmlinux alongside the .ko so kernel symbol references resolve. Without it, every call into the kernel is a black hole.
What still hurts
Indirect calls through function pointers in file_operations tables. Static analysis loses the trail; runtime tracing fills the gap.

Force-multiplier

smartdiff — semantic diff for source & binaries, via MCP

// MCP = Model Context Protocol · AST-level for source · HLIL for binaries · LLM does the reasoning
What it pairs

Functions, not bytes

AST (Abstract Syntax Tree) + semantic matching for source (Rust/C/C++/Py/JS/Java); HLIL-level matching for binaries via the Binary Ninja plugin. Survives reordering, inlining, register churn.

Two MCP servers

Old → new transport

Older flavor: Python binary_ninja_mcp plugin over SSE (Server-Sent Events). Newer Rust workspace exposes streamable HTTP MCP at /mcp (port 8011) and stdio — SSE kept only for backward compat.

What we found

Two vulnerability classes

Bug class 01 · T31

32-bit verification

  • Verifier checks one 32-bit word of the RSA output
  • RSA-2048 / SHA-256 reduced to a 32-bit brute force
  • ~217 seconds to forge an image
  • The crypto is there — just barely enforced.
Bug class 02 · T32 / T40 / T41 / A1

Init-table bypass

  • ROM parses an init table before verifying the signature
  • Init table supports full 32-bit write addresses
  • Write to SRAM → corrupt secure-boot state → skip verify
  • The verification is sound. It just never runs.
→ Different chips. Different flaws. Same outcome: unsigned code runs. coordinated disclosure in progress

Case study · T31X consumer video doorbell

T31: the 32-bit verification flaw

The Attack

Keep the header, forge the body

  1. 01Keep vendor-signed header intact
  2. 02Replace SPL (Secondary Program Loader) payload with Custom firmware
  3. 03Add a 4-byte nonce inside the payload
  4. 04Brute force until SHA-256 word[0] matches
  5. 05Flash → device accepts the forged image
The verifier checks 4 bytes of a 32-byte hash.
